# 일본 전력시장 모니터링

EPRX 조정력시장과 향후 JEPX 현물시장을 한 앱에서 구분해 분석하기 위한 Streamlit
프로젝트입니다. 현재 EPRX 1차 조정력 분석은 정상 작동하며, 실제 파일과 재현 가능한
샘플 데이터를 사용자가 직접 전환할 수 있습니다. 모든 날짜와 시간은 Asia/Tokyo
달력 기준으로 취급합니다.

## 시장 구조

### EPRX 조정력시장

실제 EPRX 파일 로드, 50Hz·60Hz 권역 비교, 도쿄·중부 상세분석, 전국 시장 요약,
데이터 품질 진단 및 규칙 기반 요약이 현재 정상 작동합니다. 내부 탭 순서는
`주파수권역 분석 → 도쿄·중부 상세분석 → 전국 시장 요약`입니다.

### JEPX 현물시장

실제 확인된 JEPX Day-Ahead CSV를 읽는 데이터 로더와 연결 상태 화면이 구현되어
있습니다. 원본 파일은 `data/jepx/raw`에 저장합니다. 현재 지원 형식은 실제 확인한
cp932·쉼표 구분 CSV이며, 임의의 Excel 형식이나 추정 열 구조는 지원하지 않습니다.

로더는 원본 19열 wide 데이터를 보존하면서 시스템가격과 9개 지역가격을 분석용
long 형식으로 변환합니다. long 데이터는 날짜, 30분 코마, Asia/Tokyo 일시, 지역,
가격, 단위, 실제 원본 열과 원본 행 번호를 포함합니다. 파일별 인코딩·열·해시·기간과
오류·검토 경고는 JEPX 화면의 `JEPX 데이터 품질 및 파일 진단`에서 확인할 수
있습니다. 실제 구조는 [JEPX_DATA_DICTIONARY.md](JEPX_DATA_DICTIONARY.md)에 기록했습니다.

JEPX 화면에는 1·2·4시간 일별 스프레드 계산 검증 기능이 있습니다. 기본 `기존 NEM
방식`은 하루 중 가장 싼 2·4·8개 30분 가격과 가장 비싼 같은 수의 가격을 각각
평균합니다. `순서 무관 연속구간`은 겹치지 않는 연속 구간 쌍을, `충전 후 방전
연속구간`은 충전 종료 뒤 방전이 시작되는 연속 구간 쌍을 비교합니다. 스프레드는
방전 평균가격에서 충전 평균가격을 뺀 값입니다.

충전·방전 코마는 겹치지 않으며 동률은 이른 period_no를 우선해 결과를 재현
가능하게 유지합니다. 48개 코마가 완전하지 않거나 중복·결측이 있으면 계산하지
않습니다. 주간·월간 통계는 완전한 일별 스프레드의 평균이며 기간 원가격의 단순
최고·최저 차이가 아닙니다. 배터리 효율, 손실, 거래 수수료, 계통비용과 열화비용은
반영하지 않으므로 결과는 이론적 가격 차이이며 실제 ESS 순수익이 아닙니다. 자세한
정의는 [JEPX_SPREAD_METHODOLOGY.md](JEPX_SPREAD_METHODOLOGY.md)를 참고하세요.

#### JEPX 주간 모니터링 사용법

JEPX 화면은 `주간 모니터링 → 계산 검증 → 데이터 품질 및 파일 진단` 순서입니다.
주간 화면에서 실제 보유 주차, 전체 지역 또는 개별 지역, 1·2·4시간 운용시간과
계산방식을 선택합니다. 기본값은 2시간과 기존 NEM 방식입니다.

주간 평균 스프레드는 완전하고 계산된 일별 결과의 산술평균입니다. 불완전한 날짜는
KPI에서 제외하고 상세표에 상태를 보존합니다. 전주 대비는 동일 지역·운용시간·
계산방식의 직전 월요일~일요일 결과와 비교하며, 전주 값이 0이면 변화율을 계산
불가로 표시합니다. 충전·방전 시간대 빈도는 시작시간의 선택 횟수이고 동률은 모두
유지합니다. 지역가격 동일 여부는 같은 날짜·30분 코마를 실제 지역가격끼리 직접
비교하며 사전에 고정한 지역 묶음을 사용하지 않습니다.

표현된 결과는 시장가격 차이이며 배터리 효율, 수수료, 계통비용 및 열화비용을
반영한 실제 ESS 순수익이 아닙니다. 월별·연도별 장기 추이와 AI 분석은 아직
구현하지 않았습니다. 세부 단계는 [JEPX_IMPLEMENTATION_PLAN.md](JEPX_IMPLEMENTATION_PLAN.md)를 참고하세요.

## Windows와 VS Code에서 실행하기

1. VS Code에서 이 프로젝트 폴더를 엽니다.
2. 메뉴에서 `터미널 > 새 터미널`을 선택합니다.
3. 가상환경을 만들고 활성화합니다.

```powershell
py -m venv .venv
.\.venv\Scripts\Activate.ps1
```

## 배포 안내

### 실행 환경

- Python: 3.14 (`runtime.txt`)
- 앱 진입점: `app.py`
- 설치: `pip install -r requirements.txt`
- 실행: `streamlit run app.py`

앱은 프로젝트 루트를 기준으로 데이터를 찾으므로 실행 위치와 무관하게 동작합니다.
EPRX 파일은 `data/eprx` 또는 `data/eprx/raw`, JEPX Day-Ahead CSV는
`data/jepx/raw`에 둡니다.

실제 EPRX·JEPX 파일은 저장소에 포함하지 않는 것이 기본 정책입니다. EPRX 화면에는
기존 재현 가능 샘플 데이터 모드가 있으며 화면 상단에 샘플임을 표시합니다. JEPX는
별도 샘플을 생성하지 않으며 파일이 없으면 한국어 안내와 빈 진단 화면을 표시합니다.

### Streamlit Community Cloud

1. 이 프로젝트를 GitHub 저장소에 업로드합니다.
2. Streamlit Community Cloud에서 저장소와 `app.py`를 선택합니다.
3. Advanced settings에서 Python 3.14를 선택합니다.
4. 비밀값이 필요한 경우 로컬 파일을 올리지 말고 Cloud의 Secrets 입력란에
   `.streamlit/secrets.toml` 형식으로 등록합니다.
5. 배포 로그에서 의존성 설치와 앱 기동을 확인합니다.

현재 앱 실행에 필수인 API 키나 비밀정보는 없습니다. 예시 형식은
`.streamlit/secrets.toml.example`에 있으며 실제 `secrets.toml`과 `.env`는 Git에서
제외됩니다.

### 현재 배포 제한사항

- 실제 시장 데이터가 Git에서 제외되면 배포 앱에는 별도 데이터 제공 절차가 필요합니다.
- EPRX 자동수집은 공식 이용 승인을 뜻하는 `EPRX_AUTOMATION_APPROVED=true`가 없는 한
  네트워크 요청 전에 중단됩니다.
- Community Cloud의 파일시스템은 영구 데이터 저장소로 사용하지 않습니다.
- 모바일·좁은 화면은 배포 전 수동 확인이 남아 있습니다.
- 상세 확인 항목은 [DEPLOYMENT_CHECKLIST.md](DEPLOYMENT_CHECKLIST.md)를 참고하세요.

PowerShell이 활성화 스크립트를 막으면 현재 터미널에서만 정책을 변경합니다.

```powershell
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
.\.venv\Scripts\Activate.ps1
```

4. 패키지를 설치하고 앱을 실행합니다.

```powershell
python -m pip install -r requirements.txt
streamlit run app.py
```

브라우저가 자동으로 열리지 않으면 터미널의 Local URL을 브라우저에 입력합니다.
종료하려면 터미널에서 `Ctrl+C`를 누릅니다.

## 실제 EPRX 파일 사용하기

1. EPRX 파일을 사용자가 직접 내려받습니다.
2. 원본을 수정하거나 Excel 형식으로 변환하지 않고 `data/eprx` 폴더 또는
   `data/eprx/raw` 폴더에 넣습니다.
3. 앱 왼쪽 사이드바에서 기본값인 `Actual EPRX files`를 선택합니다.

현재 실제로 확인하여 지원하는 형식은 **쉼표 구분 CSV**입니다. CSV 인코딩은
cp932, shift_jis, utf-8-sig, utf-8 순으로 검사하며, 일본어 메타데이터와 대체문자
여부를 확인해 정상 인코딩을 선택합니다. 현재 파일에서는 cp932가 선택되었습니다.
코드는 Excel 확장자를 발견할 수 있지만 실제 Excel 파일 구조는 확인되지 않았으며,
배포 시 파일 판독 오류가 나지 않도록 Excel 엔진 패키지는 requirements에 포함했습니다.

원본 파일의 경로·수정시각·크기가 바뀌면 Streamlit 캐시가 갱신됩니다. 앱은 원본을
읽기만 하며 수정하거나 덮어쓰지 않습니다. 여러 파일에서 동일 날짜·지역·상품·
시간대가 발견되면 삭제하지 않고 중복 후보로 표시합니다.

기존 `data/eprx` 바로 아래 파일은 자동으로 이동하지 않으며 계속 읽을 수 있습니다.
원본과 메타데이터를 구분하고 싶다면 앱을 종료한 뒤 Windows 파일 탐색기에서
사용자가 직접 원본 CSV를 `data/eprx/raw`로 이동할 수 있습니다. 같은 내용의
파일이 두 경로에 동시에 있으면 SHA-256이 같은 파일은 한 번만 분석합니다.

실제 파일을 읽지 못해도 앱은 사용자 동의 없이 샘플로 전환하지 않습니다. 화면의
`Data Quality & File Diagnostics`를 열어 다음 항목을 확인하세요.

- 파일별 성공 여부, 인코딩과 인코딩 시도 결과
- 원본·정규화·1차 조정력 행 수
- 오류 및 중복 행 수
- 필수 열 누락과 파싱 오류
- 누락 지역·시간대와 불완전 주차

파일 구조가 바뀌면 [DATA_DICTIONARY.md](DATA_DICTIONARY.md)에서 현재 확인된 원본
구조와 일본어 지표를 비교하고 `utils/eprx_loader.py`의 실제 매핑을 점검합니다.
깨져 보이는 Excel 문자열을 코드 열 이름으로 복사하지 마세요.

## 샘플 데이터 사용하기

사이드바에서 사용자가 직접 `Sample data`를 선택하면 고정 seed로 만든 최근 완료
8주 데이터를 사용합니다. 실제 파일 파싱 실패 시 자동으로 선택되지는 않습니다.

## EPRX 신규 파일 확인과 다운로드

앱 사이드바의 `EPRX 데이터 업데이트`에서 다음 기능을 제공합니다.

- `신규 파일 확인`: 링크와 신규·보유·수정 가능 후보만 확인하는 dry-run입니다.
  파일을 저장하거나 다운로드 이력을 변경하지 않습니다.
- `시험 다운로드 1개`: 확인 체크박스를 선택한 경우에만 가장 최근의 1차 조정력
  후보 한 개를 저장하고 기존 파서로 즉시 검사합니다.
- `신규 파일 다운로드 및 반영`: 이번 단계에서는 비활성화되어 있습니다. 시험
  다운로드와 파싱 검증이 완료된 이후 단계에서 사용할 예정입니다.

공식 거래실적 진입 URL은 다음과 같습니다.

```text
https://www.eprx.or.jp/information/results.php
```

EPRX 면책·이용 안내에는 자동적인 대량 취득을 하려면 사전 승인이 필요하다고
명시되어 있습니다. 따라서 이 앱은 기본 상태에서 공식 페이지 요청 전에 중단하며,
오류 안내만 표시합니다. EPRX로부터 자동수집 승인을 실제로 받은 경우에만 앱을
실행하기 전 현재 PowerShell 세션에 다음 환경변수를 설정합니다.

```powershell
$env:EPRX_AUTOMATION_APPROVED="true"
streamlit run app.py
```

환경변수는 승인을 대신하지 않습니다. 승인 없이 설정하지 마세요. 거래실적
페이지의 동의 절차, HTML 구조 또는 이용조건이 변경되면 수집 기능이 중단될 수
있으며 진단 메시지를 먼저 확인해야 합니다.

허용되는 환경변수 값은 대소문자와 관계없이 `true`, `1`, `yes`입니다. 예를 들어
`true`, `TRUE`, `True`, `1`을 사용할 수 있습니다. PowerShell에서 `$env:`로
설정한 값은 현재 터미널과 그 터미널에서 실행한 앱에만 유지됩니다. 새 터미널을
열면 다시 설정해야 합니다.

브라우저에서 완료한 동의나 승인 상태가 Python의 `requests.Session`에 자동으로
전달되는 것은 아닙니다. 사이트가 쿠키 또는 별도 세션을 요구하면 앱 요청은 다시
동의 페이지로 이동할 수 있습니다. 앱은 다음 작업을 하지 않습니다.

- 승인 폼 자동 제출
- Chrome 등 브라우저 쿠키 또는 프로필 자동 추출
- 승인 신청에 사용한 이름·연락처 등 개인정보 저장

동의 페이지, 로그인 화면, 오류 화면, HTML 문서 또는 다운로드 링크가 없는 안내
페이지가 반환되면 다운로드 전에 중단합니다.

시험 순서는 다음과 같습니다.

1. 현재 PowerShell에서 승인 환경변수를 설정합니다.
2. 앱을 실행하고 `신규 파일 확인`을 눌러 최종 URL과 페이지 판별을 확인합니다.
3. 거래실적 페이지와 1차 조정력 후보가 정상 표시되는지 확인합니다.
4. 승인 확인 체크박스를 선택합니다.
5. `시험 다운로드 1개`를 눌러 최신 후보 한 개만 다운로드합니다.
6. 파일 해시, 인코딩, 날짜 범위, 지역, 필수 6개 지표와 파싱 결과를 확인합니다.

신규 원본과 이력은 다음 위치에 저장됩니다.

```text
data/eprx/raw/                         신규 원본
data/eprx/archive/                     향후 수동 보관용(자동 이동하지 않음)
data/eprx/metadata/download_history.csv
data/eprx/metadata/update_log.csv
```

다운로드는 임시 파일에 먼저 기록하고 HTTP 상태, Content-Type, 확장자, 0바이트,
50MB 제한을 검사합니다. 완료된 파일의 SHA-256이 이력과 같으면 저장하지 않습니다.
같은 파일명이 이미 있지만 해시가 다르면 기존 파일을 덮어쓰지 않고 다음처럼
수정본 이름으로 별도 저장합니다.

```text
원본파일명__revised_YYYYMMDD_HHMMSS.csv
```

다운로드나 파싱이 실패해도 기존 데이터와 대시보드는 계속 동작합니다. 실패 원본은
삭제하거나 자동 이동하지 않으며, 앱의 업데이트 결과와 `데이터 품질 및 파일 진단`,
`download_history.csv`, `update_log.csv`의 오류 내용을 확인합니다. 두 로그에는
승인 신청자의 이름, 연락처 또는 브라우저 쿠키 같은 개인정보를 기록하지 않습니다.

## 지역과 주파수 권역

- 50Hz: Hokkaido, Tohoku, Tokyo
- 60Hz: Chubu, Hokuriku, Kansai, Chugoku, Shikoku, Kyushu

Hokkaido는 현재 50Hz에 포함하며 별도 계통 특성은 향후 검토합니다.

## 주간 집계 방식

각 지역은 하루 48개(00:00~23:30) 30분 시간대를 갖습니다. 월요일부터 일요일까지
7일의 같은 시간대를 평균해 지역별 48개 주간 프로파일을 만듭니다. 완전한 주라면
`9개 지역 × 48개 시간대 = 432행`입니다. 7일이 아닌 주차도 선택할 수 있지만
`Incomplete` 경고를 표시하고 값을 임의로 채우지 않습니다.

권역 집계 시 모집량, 입찰량, 낙찰량은 소속 지역 값을 합산합니다. 최대가격은 지역
최대가격의 최댓값, 최소가격은 지역 최소가격의 최솟값입니다. 평균가격은 아래처럼
낙찰량으로 가중평균합니다.

```text
sum(지역 평균가격 × 지역 낙찰량) / sum(지역 낙찰량)
```

완전한 권역 결과는 `2개 권역 × 48개 시간대 = 96행`입니다. 0으로 나누는 비율과
총 낙찰량이 0인 가중평균은 `NaN`으로 유지합니다. 가격·물량 단위는 원본 그대로
사용하며 임의 변환하지 않습니다.

## 분석 화면

앱은 선택 주차와 데이터 소스를 공유하는 세 개의 분석 탭을 제공합니다.

### 전국 시장 요약

일본 9개 지역의 1차 조정력 데이터를 48개 시간대별로 합산해 시장 규모,
수급, 경쟁 및 가격 참고지표를 표시합니다. 모집량·입찰량·낙찰량은 지역별
합계이며, 전국 가격 참고값은 다음과 같이 계산합니다.

```text
sum(지역 평균 낙찰가격 × 지역 낙찰량) / sum(지역 낙찰량)
```

이는 일본 전체에 공통 적용되는 단일 시장가격이 아니라 **낙찰량 가중 참고값**입니다.
최고가격은 지역 최고가격의 최댓값, 최저가격은 지역 최저가격의 최솟값을 사용합니다.
지역 누락, 48개 시간대 누락, 7일 관측 미달 및 가격 계산 불가를 검증해 경고하며,
누락값은 임의로 0으로 채우지 않습니다. 주간 물량 합계는 30분 상품구간별 MW의
합계로, MWh 에너지량이 아닙니다.

### 주파수권역 분석

기존 50Hz·60Hz 권역의 KPI, 가중평균 낙찰가격, 물량 프로파일, 입찰경쟁률과
상세 데이터를 표시합니다. 기존 계산과 필터 방식은 그대로 유지됩니다.

### 도쿄·중부 상세분석

50Hz권역의 대표 지역인 도쿄와 60Hz권역의 대표 지역인 중부를 같은 30분 시간대
기준으로 직접 비교합니다. 두 지역은 서로 다른 주파수권역에 속하므로 권역 집계
전의 지역별 가격·물량·경쟁 정도 차이를 확인할 수 있습니다.

지역별 7일 데이터를 같은 시간대끼리 평균해 도쿄 48행, 중부 48행을 만듭니다.
물량 KPI는 48개 시간대 평균이며, 입찰경쟁률·조달률·입찰 대비 낙찰률은 기간
합계끼리 나눕니다. 평균 낙찰가격은 선택 주 원본 가격을 낙찰량으로 가중평균합니다.

가격 범위는 각 시간대의 `최고 낙찰가격 − 최저 낙찰가격`입니다. 미조달량은
`모집량 − 낙찰량`과 0 중 큰 값으로 계산합니다. 조달률이 100%를 초과해도 원본
공표값을 임의로 제한하지 않습니다.

전주 대비 표는 선택 주와 직전 보유 주차의 현재 값, 전주 값, 절대 변화 및 변화율을
보여줍니다. 전주 값이 0이면 변화율은 계산 불가로 유지하고, 전주 데이터가 7일
미만이면 경고를 표시합니다.

상단 자동 요약은 계산 결과를 조건문으로 문장화한 **규칙 기반 설명**입니다.
AI 예측이나 투자·사업성 판단이 아니며, 가격·입찰경쟁률·미조달·전주 변화 같은
관측 결과만 객관적으로 설명합니다.
